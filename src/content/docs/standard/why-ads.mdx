## The Problem

Architecture documentation is inconsistent across the industry. Teams use a patchwork of templates — some too abstract to guide real work, others too platform-specific to generalise. After analysing eight widely-used frameworks and real-world architecture documents, gaps appear in **every single one**.

## Landscape Analysis

We reviewed publicly available architecture documentation frameworks, templates, and filled-in examples across open-source, enterprise, government, and EU research contexts.

### Frameworks and Templates Reviewed

| Framework / Template | Origin | Strengths | Key Gaps |
|---|---|---|---|
| **arc42** | Open-source (Europe) | Most complete open template; 35+ real examples; multi-level building block views; strong community | No Security View; no Data View; no cloud WAF-aligned quality attributes; no depth levels |
| **bflorat template** | French government / enterprise | Role-based view separation; consistent Constraints → NFRs → Solution structure per view; dedicated Security View | No Integration View; no Scenarios View; no lifecycle; no risk governance; no ADRs |
| **shekhargulati template** | GitHub (startups) | Back-of-envelope sizing calculations; lightweight and accessible | No Physical, Security, Data, or Process views; no quality attributes; no lifecycle |
| **ServiceNow SAD** | Platform-specific (ITSM) | Strong data governance (classification, archival, anonymisation); current-to-future-state progression | Platform-specific; monolithic structure; no architectural views; no quality attributes |
| **NHS England SAF** | UK Government (NHS) | Governance-focused; references cloud WAFs; RAID-based risk tracking | Meta-framework only — describes what to do but provides no concrete fill-in template |
| **HELMET D3.1 HLD** | EU Horizon 2020 (transport) | Real 86-page filled-in HLD; multi-domain (rail, automotive, UAV); safety-integrity focus | Organised by domain segment not viewpoints; no Security View; no quality attributes |
| **ALMBoK SAD** | Enterprise methodology | 13 viewpoints — broadest coverage found; full As-Is to To-Be lifecycle | 13 viewpoints with no guidance on which to skip; no depth levels; no schema; no atomic fields |
| **nocomplexity Playbook** | Open-source | Uniquely addresses relations among views | Too abstract — doesn't prescribe which views to include; no quality attributes; skeletal |

### The Five Universal Gaps

Every framework analysed was missing at least five of these capabilities:

## How ADS Differs from ISO 42010

ADS is *aligned* with ISO/IEC/IEEE 42010:2022 but goes substantially beyond it:

- **Quality Attributes** — ISO 42010 defines viewpoints and views but does not prescribe quality assessment. ADS incorporates operational excellence, reliability, performance efficiency, cost optimisation, and sustainability as cross-cutting quality attributes mapped to AWS, Azure, GCP, Oracle, and IBM Well-Architected Frameworks.
- **Documentation Depth Levels** — ISO 42010 does not address how much detail is needed. ADS defines three tiers (Minimum / Recommended / Comprehensive) with RFC 2119 requirement keywords (SHALL / SHOULD / MAY).
- **Security as a First-Class View** — ISO 42010 does not prescribe a security viewpoint. ADS includes a dedicated Security View (Section 3.5) covering threat modelling, identity, network security, data protection, and compliance.
- **Lifecycle Management** — ISO 42010 focuses on the architecture description itself, not the solution lifecycle. ADS includes CI/CD pipeline design, operational runbooks, and exit/decommissioning planning.
- **Decision Making & Governance** — ISO 42010 does not address risk registers, compliance traceability, or architecture decision records. ADS includes all three.
- **JSON Schema** — ISO 42010 is a prose standard. ADS provides a machine-readable JSON Schema with atomic fields for automated validation and multi-format generation.
- **Downloadable Templates** — ISO 42010 provides no templates. ADS includes ready-to-use Markdown, YAML, and JSON templates.
