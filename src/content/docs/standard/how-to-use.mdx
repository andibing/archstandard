## 2.1 Terminology

The key words "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "MAY", and "OPTIONAL" in this standard are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174).

| Keyword | Meaning |
|---------|---------|
| **SHALL** | An absolute requirement of the standard |
| **SHALL NOT** | An absolute prohibition |
| **SHOULD** | Recommended; there may be valid reasons to deviate but the implications must be understood |
| **SHOULD NOT** | Not recommended; there may be valid reasons to include but the implications must be understood |
| **MAY** / **OPTIONAL** | Truly optional; include at the author's discretion |

These keywords map to the standard's documentation depths as follows:

| Documentation Depth | Keyword | Governance Gate |
|---------------|---------|-----------------|
| <span class="maturity-indicator maturity-minimum">Minimum</span> | SHALL | Development / Test review |
| <span class="maturity-indicator maturity-recommended">Recommended</span> | SHOULD | Production approval |
| <span class="maturity-indicator maturity-comprehensive">Comprehensive</span> | MAY | Enterprise review |

## 2.2 Adopting the Standard

### Step 1: Determine Your Maturity Target

Select the documentation depth appropriate for your project phase and governance requirements:

- **Minimum** — For early-stage designs, proofs of concept, or development/test architecture reviews. Complete all sections marked <span class="maturity-indicator maturity-minimum">Minimum</span>.
- **Recommended** — For production-bound designs requiring formal governance approval. Complete all sections marked <span class="maturity-indicator maturity-minimum">Minimum</span> and <span class="maturity-indicator maturity-recommended">Recommended</span>.
- **Comprehensive** — For critical, regulated, or enterprise-scale systems. Complete all sections.

### Step 2: Complete the Relevant Sections

Work through each section of the standard in order. Sections above your target documentation depth SHOULD be **omitted entirely** rather than included with empty or "Not Applicable" content. This keeps the document focused and avoids template bloat.

Where a section at or below your target depth is genuinely not relevant to the solution (e.g., IoT Devices for a purely cloud-based API), it SHOULD also be omitted. Only include sections that add value to the reader.

### Step 3: Evaluate Against Quality Attributes

After completing the architectural views (Section 3), evaluate the design against each Quality Attribute (Section 4):

1. For each quality attribute, assess whether the design adequately addresses the concerns
2. Document any tradeoffs between quality attributes and the rationale for the chosen balance
3. Cross-reference specific quality attribute requirements from within the architectural views

### Step 4: Validate and Submit for Review

1. If using machine-readable output, validate the document against the JSON Schema (see [Schema](/standard/schema/))
2. Submit for governance review at the appropriate gate for your documentation depth

## 2.3 Document Lifecycle

A Solution Architecture Document produced to this standard is a **living artefact**:

- It SHALL describe the **current state** of the solution architecture
- It SHALL be updated when architecturally significant changes are made
- It MAY be updated by multiple teams or change activities, either sequentially or in parallel
- It SHALL NOT be owned by a single project team; ownership rests with the accountable architect for the solution
- Changes to the design SHOULD be approved through the organisation's architecture governance process

<Aside type="tip" title="Incremental Updates">
If a project seeks only to make incremental changes to a solution documented in an existing HLD, only the sections relevant to the change need to be updated. The remaining sections SHALL remain unchanged and continue to describe the current state.
</Aside>

## 2.4 Organisation Customisation

This standard is designed to be adopted without modification. However, organisations MAY extend it in the following ways:

1. **Organisation Profile** — Map generic sections to specific internal tools, standards, and governance processes using the `organisationProfile` field in the JSON Schema
2. **Custom Sections** — Add organisation-specific sections using the `customSections` extension point without modifying the core standard structure
3. **Standards Traceability** — Reference internal design principles, patterns, and standards within the compliance traceability section (Section 6.4)
4. **Governance Gate Mapping** — Map the three documentation depths to your organisation's architecture governance stages (e.g., ARB, design authority)

Organisations SHALL NOT renumber core sections of the standard. Sections that are not applicable to the solution SHOULD be omitted entirely to keep documents focused and readable.

## 2.5 Output Formats

The standard's JSON Schema enables generation and validation in multiple formats:

| Format | Use Case |
|--------|----------|
| **Web** | Interactive documentation site (this site) |
| **Markdown** | Inclusion in git repositories alongside code |
| **Word / DOCX** | Formal governance submissions and offline review |
| **JSON** | Machine-readable for tooling integration and validation |

See [Templates](/standard/templates/) for downloadable blank templates in each format.
