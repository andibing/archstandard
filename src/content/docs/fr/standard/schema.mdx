## Vue d'ensemble

L'ADS définit un **JSON Schema** formel (v1.0.0) qui permet :

- **Validation** - Valider les documents d'architecture de solution par rapport à la structure standard
- **Génération** - Générer des modèles vierges dans plusieurs formats (JSON, YAML, Markdown, Word)
- **Outillage** - Créer des outils qui lisent, écrivent et traitent les documents d'architecture de solution de manière programmatique
- **Extensibilité** - Ajouter des sections spécifiques à l'organisation via `customSections` et `organisationProfile`

## Emplacement du schéma

Le fichier de schéma se trouve à [`/schema/ads.schema.json`](https://github.com/andibing/archstandard/blob/main/schema/ads.schema.json) dans le dépôt.

**Schema ID :** `https://archstandard.org/schema/v1.0.0/ads.schema.json`

## Structure du schéma

Le schéma correspond directement aux sections du standard :

```
ads.schema.json
├── schemaVersion          "1.0.0"
├── documentControl        Section 0: metadata, history, contributors
├── executiveSummary       Section 1: overview, context, scope, decisions
├── stakeholders           Section 2: register, concerns, compliance
├── architecturalViews     Section 3
│   ├── logicalView        3.1: components, services, patterns
│   ├── integrationView     3.2: connectivity, integrations, APIs
│   ├── physicalView       3.3: hosting, compute, networking, environments
│   ├── dataView           3.4: data stores, classification, privacy
│   ├── securityView       3.5: IAM, encryption, monitoring
│   └── scenarios          3.6: use cases, ADRs
├── qualityAttributes      Section 4: quality attributes + tradeoffs
├── lifecycleManagement    Section 5: SDLC, operations, exit planning
├── riskGovernance         Section 6: constraints, risks, decisions, compliance
├── appendices             Section 7: glossary, references, approvals
├── organisationProfile    Organisation-specific configuration
└── customSections[]       Extension point for additional sections
```

## Caractéristiques principales du schéma

### Niveaux de documentation

Les sections utilisent un enum `maturityLevel` avec trois valeurs. Note : le schéma utilise le nom de propriété `maturityLevel` tandis que la prose du standard désigne ce concept par « Documentation Depth » — ils signifient la même chose :

```json
{ "type": "string", "enum": ["minimum", "recommended", "comprehensive"] }
```

### Références aux attributs de qualité

Toute section peut référencer les attributs de qualité applicables :

```json
{
  "qualityAttributeRefs": ["reliability", "performance", "cost-optimisation"]
}
```

### Profil d'organisation

L'extension `organisationProfile` permet de mapper le standard générique aux outils et à la gouvernance spécifiques de l'organisation :

```json
{
  "organisationProfile": {
    "organisationName": "Acme Corp",
    "tooling": {
      "cmdb": "ServiceNow",
      "secretStore": "HashiCorp Vault",
      "monitoring": "Datadog"
    },
    "governanceGates": [
      {
        "gateName": "Dev/Test Review",
        "requiredMaturityLevel": "minimum",
        "requiredSections": ["documentControl", "executiveSummary", "architecturalViews.logicalView"]
      },
      {
        "gateName": "Production Approval",
        "requiredMaturityLevel": "recommended"
      }
    ]
  }
}
```

### Sections personnalisées

Ajoutez des sections spécifiques à l'organisation sans modifier le schéma de base :

```json
{
  "customSections": [
    {
      "id": "cloud-governance",
      "title": "Cloud Governance",
      "description": "Organisation-specific cloud governance requirements",
      "content": "...",
      "maturityLevel": "recommended",
      "parentSection": "physicalView"
    }
  ]
}
```

## Champs obligatoires et facultatifs

Le schéma n'impose comme obligatoires que les champs du niveau de documentation **Minimum** :

| Champ obligatoire | Section |
|-------------------|---------|
| `schemaVersion` | Racine |
| `documentControl.metadata` | 0. Document Control |
| `executiveSummary.solutionOverview` | 1. Executive Summary |
| `architecturalViews` | 3. Architectural Views |
| `riskGovernance.risks` | 6. Decision Making & Governance |

Tous les autres champs sont facultatifs, permettant une adoption progressive du niveau Minimum au niveau Comprehensive.

## Exemple de validation

Validez un SAD par rapport au schéma en utilisant des références locales ou distantes :

```bash
# Using ajv-cli (local schema)
npx ajv validate -s schema/ads.schema.json -d my-sad.json

# Using ajv-cli (remote schema)
npx ajv validate -s https://archstandard.org/schema/v1.0.0/ads.schema.json -d my-sad.json

# Using Python jsonschema
pip install jsonschema
python -m jsonschema -i my-sad.json schema/ads.schema.json
```

Le schéma est hébergé à : [`https://archstandard.org/schema/v1.0.0/ads.schema.json`](https://archstandard.org/schema/v1.0.0/ads.schema.json)

## Créer des outils avec le schéma

Le JSON Schema est conçu pour être consommé par l'outillage — constructeurs de formulaires, validateurs, convertisseurs et plateformes de gestion de SAD. Les propriétés d'extension personnalisées facilitent cela :

| Extension | Objectif | Exemple |
|-----------|----------|---------|
| `x-ads-section` | Numéro de section | `"3.1"`, `"6.3"` |
| `x-ads-title` | Nom d'affichage | `"Logical View"`, `"Risks"` |
| `x-ads-depth` | Niveau de documentation | `"minimum"`, `"recommended"`, `"comprehensive"` |

Ces extensions sont présentes sur chaque section et sous-section majeure du schéma. Un outil peut parcourir l'arborescence du schéma et utiliser ces propriétés pour :

- **Générer des formulaires** avec le regroupement correct des sections, les libellés et les indicateurs de niveau
- **Afficher/masquer des sections** en fonction du niveau de documentation choisi par l'utilisateur
- **Valider la complétude** par rapport à un niveau spécifique
- **Exporter** en JSON, YAML, Markdown ou DOCX

```javascript
// Example: fetch the schema and list all sections
const schema = await fetch('https://archstandard.org/schema/v1.0.0/ads.schema.json').then(r => r.json());

for (const [key, def] of Object.entries(schema.$defs)) {
  if (def['x-ads-section']) {
    console.log(`${def['x-ads-section']} ${def['x-ads-title']} [${def['x-ads-depth']}]`);
  }
}
```
