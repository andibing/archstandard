## Conception à champs atomiques

ADS est conçu avec des **champs atomiques** — des valeurs structurées et discrètes plutôt que du texte libre autant que possible. C'est un principe fondamental du standard.

### Pourquoi des champs atomiques ?

| Avantage | Description |
|---------|-------------|
| **Réduction de l'ambiguïté** | Les auteurs sélectionnent parmi des options définies plutôt que d'inventer de la prose |
| **Traitement automatisé** | Les outils peuvent agréger, comparer, valider et générer des rapports à travers les SAD |
| **Plus rapide à compléter** | Les cases à cocher et les sélections sont plus rapides que les champs de texte vierges |
| **Comparable** | Deux SAD peuvent être structurellement comparés et différenciés |
| **Traduisible** | Les valeurs enum peuvent être associées à n'importe quelle langue sans ambiguïté |
| **Prêt pour la gouvernance** | Le processus de gouvernance architecturale peut vérifier la complétude de manière programmatique |

### Comment cela fonctionne selon les formats

Le même champ atomique s'affiche différemment selon le format de sortie, mais porte une signification identique :

| Type de schéma | JSON / YAML | Markdown / Word | Confluence / Wiki |
|------------|-------------|-----------------|-------------------|
| `boolean` | `true` / `false` | Case à cocher : ☑ / ☐ | Macro de case à cocher |
| `enum` (simple) | `"active-active"` | Bouton radio ou sélection en gras | Macro de statut ou liste déroulante |
| `enum` (tableau) | `["aws", "azure"]` | Cases à cocher multiples : ☑ AWS ☑ Azure | Macro de sélection multiple |
| `yesNoNa` | `"yes"` / `"no"` / `"not-applicable"` | ☑ Yes ☐ No ☐ N/A | Macro de statut |
| `riskLevel` | `"high"` | **HIGH** (avec couleur) | Pastille de statut |
| `integer` / `number` | `4096` | Champ numérique | Champ numérique |
| `string` (contraint) | `"PT4H"` (ISO 8601) | « 4 heures » | Champ de durée |
| `string` (texte libre) | Prose | Prose | Prose |

<Aside type="tip" title="Champs de texte libre">
Les champs de texte libre sont toujours utilisés lorsque les options structurées ne sont pas praticables (par ex., `solutionOverview`, `mitigationPlan`, `rationale`). Cependant, même ceux-ci sont encadrés — ils ont un objectif spécifique décrit dans le schéma, pas des champs de « notes » ouverts.
</Aside>

### Catégories de champs atomiques dans ADS

Le schéma utilise les patterns atomiques suivants :

| Pattern | Exemples de champs | Valeurs |
|---------|---------------|--------|
| **Yes / No / N/A** | `internetFacing`, `thirdPartyHosted`, `backupEnabled` | `yes`, `no`, `not-applicable` |
| **Risk Level** | `residualRisk`, `confidentiality`, `integrity` | `critical`, `high`, `medium`, `low`, `negligible` |
| **Classification** | `classification` | `public`, `internal`, `restricted`, `highly-restricted` |
| **Status** | `document status`, `ADR status`, `risk status` | Définis selon le contexte |
| **Technology Enum** | `protocol`, `authenticationMethod`, `secretStore` | Définis par champ |
| **Sizing Category** | `dataSizeCategory`, `clusterSize` | Plages catégorisées |
| **Duration** | `retentionPeriod`, `rtoTarget` | ISO 8601 ou catégories par plages |
| **Criticality** | `businessCriticality` | `tier-1-critical` à `tier-5-minimal` |
| **ID Pattern** | `UC-01`, `ADR-001`, `R-001` | Chaînes contraintes par regex |

## Compatibilité multi-format

ADS est conçu pour fonctionner aussi bien comme :

- **Un document rédigé par un humain** (Word, Confluence, wiki) — où les champs atomiques deviennent des cases à cocher, des menus déroulants et des boutons radio
- **Un fichier lisible par machine** (JSON, YAML, Markdown) — où les champs atomiques sont des enums, des booléens et des valeurs typées
- **Un formulaire web** — où les champs atomiques correspondent directement à des contrôles de formulaire

Le JSON Schema est la source unique de vérité. Tous les formats de sortie en sont dérivés, garantissant la cohérence quel que soit le mode de rédaction ou de consultation du SAD.

## Accessibilité et charge cognitive

ADS est conçu pour être utilisable par tous les architectes, y compris les personnes neurodivergentes. Les principes suivants réduisent la charge cognitive et améliorent la lisibilité :

### Cohérence structurelle

Chaque section du standard suit le même schéma :

1. **Objectif** — une courte déclaration expliquant ce que couvre cette section et pourquoi
2. **Tableaux / Champs** — des champs structurés et atomiques à compléter
3. **Recommandations** — une aide contextuelle dans un encadré clairement identifié

Cette structure prévisible signifie que vous savez toujours à quoi vous attendre en ouvrant une nouvelle section.

### Réduction de la charge cognitive

| Principe | Comment ADS l'implémente |
|-----------|----------------------|
| **Champs atomiques plutôt que pages vierges** | Les cases à cocher, enums et tableaux structurés remplacent les champs de texte ouverts. Vous sélectionnez plutôt que vous composez. |
| **Divulgation progressive** | Trois niveaux de documentation (Minimum → Recommandé → Complet) vous permettent de commencer simplement et d'ajouter des détails uniquement quand nécessaire. |
| **Une idée par ligne** | Les tableaux présentent un champ par ligne plutôt que de combiner plusieurs concepts. |
| **Terminologie cohérente** | Le [glossaire](/fr/standard/7-appendices/#71-glossaire) définit tous les acronymes. Les termes sont utilisés de manière cohérente tout au long du document. |
| **Marqueurs visuels** | Des indicateurs de niveau codés par couleur (<span class="maturity-indicator maturity-minimum">Minimum</span> <span class="maturity-indicator maturity-recommended">Recommandé</span> <span class="maturity-indicator maturity-comprehensive">Complet</span>) vous permettent de voir rapidement ce qui est requis. |
| **Scoring clair** | L'échelle de conformité 0–5 fournit un retour numérique non ambigu plutôt qu'une prose subjective. |

### Acronymes et terminologie

Tous les acronymes utilisés dans ADS sont définis dans le [glossaire](/fr/standard/7-appendices/#71-glossaire). Lors de la rédaction d'un SAD :

- Définissez tous les acronymes à leur première utilisation dans le document
- Incluez un glossaire (Section 7.1) dans chaque SAD
- Évitez le jargon lorsqu'une alternative en langage courant existe

### Directives de lisibilité pour les auteurs de SAD

Lors de la rédaction de contenu en texte libre dans un SAD :

- Utilisez des phrases courtes. Une idée par phrase.
- Utilisez des listes à puces plutôt que de longs paragraphes.
- Utilisez la voix active (« Le service envoie des données vers... ») et non la voix passive (« Les données sont envoyées par le service vers... »).
- Utilisez des diagrammes pour compléter le texte, pas pour le remplacer.
- Assurez-vous que les diagrammes ont des descriptions de texte alternatif pour les lecteurs d'écran.

## Extensibilité sans ambiguïté

Lorsque les organisations ajoutent des sections personnalisées via le point d'extension `customSections`, elles SHOULD suivre les mêmes principes de champs atomiques :

1. Utilisez `boolean` pour les questions oui/non
2. Définissez des valeurs `enum` pour tout champ avec un ensemble fini de réponses valides
3. Utilisez des champs typés (`integer`, `number`, `date`) pour les valeurs mesurables
4. Réservez les champs `string` en texte libre pour le contenu véritablement ouvert
5. Ajoutez `description` à chaque champ pour clarifier son objectif
