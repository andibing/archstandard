## Objectif

Les scénarios sont le « +1 » dans le modèle de vues 4+1. Ils servent de tissu conjonctif entre toutes les autres vues, validant que l'architecture fonctionne de bout en bout pour les cas d'utilisation clés. En termes ISO 42010, ils documentent les **correspondances** entre les vues.

Cette section capture également les enregistrements de décisions architecturales (ADR) — les décisions significatives qui ont façonné la conception et leur justification.

## 3.6.1 Cas d'utilisation clés

<span class="maturity-indicator maturity-recommended">Recommandé</span>

Documentez les cas d'utilisation les plus significatifs sur le plan architectural. Chaque cas d'utilisation devrait traverser plusieurs vues pour valider la conception :

### Modèle de cas d'utilisation

**UC-[nn] : [Nom du cas d'utilisation]**

| Attribut | Détail |
|-----------|--------|
| **Acteur(s)** | *[qui initie ce cas d'utilisation]* |
| **Déclencheur** | *[ce qui démarre le flux]* |
| **Pré-conditions** | *[état requis avant l'exécution]* |
| **Flux principal** | *[description étape par étape]* |
| **Post-conditions** | *[état après exécution réussie]* |
| **Vues impliquées** | Logical / Integration & Data Flow / Physical / Data / Security |

<div class="guidance-box">
<h4>Recommandations</h4>

Sélectionnez 3 à 5 cas d'utilisation les plus significatifs sur le plan architectural :
- Le parcours utilisateur principal (chemin nominal)
- Un flux à haut volume ou critique en termes de performance
- Un scénario de panne/restauration
- Un scénario pertinent pour la sécurité (par ex., authentification, accès aux données)
- Un scénario riche en intégrations

Chaque cas d'utilisation devrait démontrer comment les composants de la vue logique communiquent (vue intégration et flux de données), sont déployés (vue physique), gèrent les données (vue données) et sont sécurisés (vue sécurité).
</div>

## 3.6.2 Enregistrements de décisions architecturales (ADR)

<span class="maturity-indicator maturity-recommended">Recommandé</span>

Documentez les décisions architecturales clés en utilisant le format ADR :

### Modèle ADR

**ADR-[nnn] : [Titre de la décision]**

| Champ | Contenu |
|-------|---------|
| **Statut** | Proposed / Accepted / Superseded / Deprecated |
| **Date** | *[date de la décision]* |
| **Contexte** | *[quel est le problème ou la question]* |
| **Décision** | *[ce qui a été décidé]* |
| **Alternatives considérées** | *[quelles autres options ont été évaluées]* |
| **Conséquences** | *[implications positives et négatives]* |
| **Compromis sur les attributs de qualité** | *[quels attributs de qualité sont affectés et comment]* |

<div class="guidance-box">
<h4>Recommandations</h4>

Enregistrez les décisions qui :
- Sont difficiles ou coûteuses à inverser
- Impliquent des compromis significatifs entre les attributs de qualité
- Dévient des patterns ou politiques standards
- Ont été débattues ou pour lesquelles plusieurs approches valides existaient
- Affectent plusieurs vues ou parties prenantes

Les ADR devraient être immuables une fois acceptés. Si une décision est inversée, créez un nouvel ADR qui remplace l'original plutôt que de le modifier.
</div>

### Exemples d'ADR à considérer

| Catégorie | Exemple de décision |
|----------|-----------------|
| Plateforme | Choix du fournisseur cloud, site d'hébergement, modèle de service |
| Architecture | Monolithe vs. microservices, événementiel vs. requête-réponse |
| Données | Choix de la technologie de base de données, stratégie de cache |
| Sécurité | Approche d'authentification, standards de chiffrement |
| Intégration | Synchrone vs. asynchrone, style d'API |
| Résilience | Active-active vs. active-passive, stratégie DR |

<div class="guidance-box">
<h4>Guide de scoring</h4>

| Score | À quoi cela ressemble |
|:-----:|---------------------|
| **1** | Un ou deux cas d'utilisation esquissés ; pas d'ADR |
| **3** | Cas d'utilisation clés documentés avec acteurs et flux ; décisions significatives capturées en ADR avec justification |
| **5** | Tout ce qui précède plus les cas d'utilisation référencent toutes les vues pertinentes ; les ADR incluent les alternatives considérées et l'impact sur les attributs de qualité |
</div>
