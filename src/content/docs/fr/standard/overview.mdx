## Objectif

Ce standard définit la structure et le contenu requis pour les **documents d'architecture de solution (SAD)**. Les vues architecturales et les attributs de qualité au sein d'un SAD constituent le **High Level Design (HLD)** de la solution.

Un document d'architecture de solution conforme à ce standard remplit les fonctions suivantes :

1. **Communication** — Fournit une compréhension partagée de l'architecture de la solution entre toutes les parties prenantes
2. **Gouvernance** — Permet au processus de gouvernance architecturale d'évaluer la conception par rapport aux critères de qualité
3. **Traçabilité** — Établit des liens entre les décisions de conception, les exigences métier, les attributs de qualité et les standards applicables
4. **Référence** — Sert de document vivant décrivant fidèlement l'architecture en l'état actuel
5. **Preuve** — Démontre que la solution est bien architecturée par rapport aux cadres de référence reconnus par l'industrie

## Périmètre

Ce standard s'applique à la documentation de toute architecture de solution technologique, indépendamment de :

- **Modèle de déploiement** — cloud-native, on-premises, hybride, multi-cloud ou SaaS
- **Échelle** — d'applications individuelles à des écosystèmes multi-applications
- **Secteur** — le standard est indépendant de l'organisation et neutre vis-à-vis du secteur
- **Phase du cycle de vie** — nouvelles constructions, migrations, modernisations ou changements incrémentaux

## Audience

Ce standard est destiné à être utilisé par :

| Rôle | Sections principales |
|------|-----------------|
| **Architectes de solution** | Toutes les sections — auteurs principaux des SAD |
| **Architectes d'entreprise** | Alignement des cadres, Attributs de qualité, Gouvernance |
| **Architectes sécurité** | Section 3.5 (Vue sécurité), Attributs de qualité |
| **Ingénieurs infrastructure** | Section 3.3 (Vue physique), Attributs de qualité |
| **Responsables développement** | Section 3.1 (Vue logique), Section 5 (Cycle de vie) |
| **Gouvernance architecturale** | Toutes les sections — comme cadre de revue et de gouvernance (par ex., ARB, autorité de conception) |

## Structure du document

Un document d'architecture de solution conforme à ADS est organisé en trois niveaux :

### Niveau 1 — Vues architecturales (Sections 3.1 – 3.6)

Six vues complémentaires décrivent l'architecture depuis des perspectives distinctes des parties prenantes, alignées sur le modèle de vues architecturales 4+1 de Kruchten (étendu avec des vues dédiées Données et Sécurité) et le concept de point de vue ISO/IEC 42010.

Ensemble, ces vues forment le **High Level Design (HLD)** de la solution :

| Vue | Point de vue | Parties prenantes principales |
|------|-----------|---------------------|
| **3.1 Vue logique** | Architecture applicative, composants, patterns | Architectes, Développeurs |
| **3.2 Vue intégration et flux de données** | Flux de données, intégrations, interfaces | Intégrateurs, Architectes |
| **3.3 Vue physique** | Déploiement, hébergement, réseau, environnements | Infrastructure, DevOps |
| **3.4 Vue données** | Stockages de données, classification, confidentialité, cycle de vie | Architectes données, Conformité |
| **3.5 Vue sécurité** | IAM, chiffrement, surveillance, modélisation des menaces | Sécurité, RSSI, Conformité |
| **3.6 Scénarios** | Cas d'utilisation clés, enregistrements de décisions architecturales | Toutes les parties prenantes |

Aucune vue à elle seule ne fournit une description complète de l'architecture. Ensemble, les six vues offrent une vision holistique répondant à toutes les préoccupations des parties prenantes.

### Niveau 2 — Attributs de qualité (Sections 4.1 – 4.5)

Cinq attributs de qualité transversaux sont évalués **à travers** toutes les vues architecturales. Ils sont dérivés des frameworks Well-Architected AWS et Azure :

| Attribut de qualité | Focus |
|--------|-------|
| **4.1 Excellence opérationnelle** | Observabilité, surveillance, procédures opérationnelles |
| **4.2 Fiabilité et résilience** | DR, scalabilité, tolérance aux pannes, sauvegarde et restauration |
| **4.3 Efficacité des performances** | Exigences de performance, optimisation des ressources |
| **4.4 Optimisation des coûts** | Analyse des coûts, pratiques FinOps |
| **4.5 Durabilité** | Efficacité énergétique, impact carbone, efficacité des ressources |

### Niveau 3 — Cycle de vie et gouvernance (Sections 5 – 7)

Les sections complémentaires traitent de la manière dont la solution est développée, opérée, gouvernée et retirée tout au long de son cycle de vie.

### Comment les sections sont reliées

Les attributs de qualité évaluent *la qualité* de ses performances. Le cycle de vie et la gouvernance couvrent *comment* elle est construite, exploitée et gouvernée.

## Niveaux de documentation

Chaque section de ce standard se voit attribuer un niveau de documentation — <span class="maturity-indicator maturity-minimum">Minimum</span> (SHALL), <span class="maturity-indicator maturity-recommended">Recommandé</span> (SHOULD), ou <span class="maturity-indicator maturity-comprehensive">Complet</span> (MAY) — qui définit quand elle doit être complétée. Cela permet une adoption progressive : commencez par le Minimum pour les conceptions en phase initiale, puis étendez au Complet pour les systèmes critiques et réglementés.

Voir [Conformité et utilisation](/v1/fr/standard/how-to-use/) pour le tableau complet des niveaux, les définitions de la terminologie et la correspondance avec les portes de gouvernance.

## Conformité

Un document d'architecture de solution **est conforme** à ce standard lorsque :

1. Toutes les sections marquées <span class="maturity-indicator maturity-minimum">Minimum</span> sont complétées
2. La structure du document suit la numérotation et la hiérarchie des sections définies dans le présent document
3. Les vues architecturales répondent aux préoccupations des parties prenantes identifiées dans la Section 2
4. Les attributs de qualité sont évalués comme des perspectives transversales à travers les vues architecturales
5. Le document est validé par rapport au JSON Schema (ADS Schema v1.0.0) si une sortie lisible par machine est requise

Un document qui complète en outre toutes les sections <span class="maturity-indicator maturity-recommended">Recommandé</span> atteint la **conformité complète**.
