## What is ADS-001?

ADS-001 is a **prescriptive, template-level standard** that defines exactly what sections, content, and quality criteria a Solution Architecture Document (SAD) shall contain.

It is built upon — but is not a replica of — ISO/IEC/IEEE 42010, which defines an abstract conceptual model for architecture descriptions. Where ISO 42010 says *"an architecture description shall identify stakeholders and their concerns"*, ADS-001 says *"Section 2.1 shall contain a stakeholder register table with columns for stakeholder, role, key concerns, and relevant architectural views."*

### The Naming Model

| Term | What It Means | In This Standard |
|------|--------------|-----------------|
| **Architecture Description** | The ISO 42010 formal term for the work product | The standard itself (ADS-001) aligns to this concept |
| **Solution Architecture Document (SAD)** | The enterprise-practice name for the document | The document you produce using this standard |
| **High Level Design (HLD)** | The conceptual-level design content | Sections 3–4 (Architectural Views + Quality Pillars) within the SAD |

## Scope

This standard defines the required and recommended structure for Solution Architecture Documents that are:

- **Organisation-agnostic** — adoptable by any enterprise without modification
- **Platform-neutral** — applicable to cloud, on-premises, hybrid, and SaaS architectures
- **Framework-aligned** — traceable to ISO/IEC 42010, the 4+1 View Model, and the AWS/Azure Well-Architected Frameworks
- **Schema-driven** — backed by a formal JSON Schema enabling validation and multi-format generation

## Beyond ISO/IEC 42010

ADS-001 *conforms to* the ISO 42010 conceptual model while providing the concrete, actionable structure that ISO 42010 deliberately leaves to implementers. It goes well beyond ISO 42010 by incorporating:

- **Quality Pillars from AWS/Azure Well-Architected Frameworks** — Operational Excellence, Reliability, Performance Efficiency, Cost Optimisation, and Sustainability as cross-cutting evaluation lenses applied across all architectural views
- **Maturity Levels with RFC 2119 requirement keywords** — every section is tagged as SHALL (Minimum), SHOULD (Recommended), or MAY (Comprehensive), enabling progressive adoption from lightweight designs to full enterprise documentation
- **Security as a dedicated Architectural View** — ISO 42010 does not prescribe specific viewpoints; ADS-001 elevates Security to a first-class view covering IAM, encryption, threat modelling, and monitoring
- **Lifecycle Management** — CI/CD pipelines, service transition, release management, operations, maintainability, and exit planning
- **Risk & Governance with compliance traceability** — design risks, guardrail exceptions, architecture decision records, and a traceability matrix linking design elements to the standards they satisfy
- **JSON Schema for multi-format generation** — a formal schema enabling validation and generation of Solution Architecture Documents in JSON, YAML, Markdown, or Word
- **Downloadable templates** — ready-to-use blank templates in Markdown, YAML, and JSON formats

### Comparison with ISO/IEC 42010

| Aspect | ISO/IEC/IEEE 42010:2022 | ADS-001 (This Standard) |
|--------|------------------------|------------------------|
| **Type** | Meta-standard (conceptual model) | Prescriptive standard (template-level) |
| **Prescribes sections?** | No — defines concepts only | Yes — defines exact sections, tables, and content |
| **Prescribes quality criteria?** | No | Yes — five quality pillars from AWS/Azure WAF |
| **Provides templates?** | No | Yes — Markdown, YAML, JSON templates |
| **Defines maturity levels?** | No | Yes — Minimum, Recommended, Comprehensive |
| **Includes security view?** | Not specifically | Yes — full dedicated Security View |
| **Includes lifecycle?** | No | Yes — CI/CD, operations, exit planning |
| **JSON Schema?** | No | Yes — machine-readable validation |

## Terms and Definitions

The key words "SHALL", "SHOULD", "MAY", and "OPTIONAL" in this standard are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174):

| Maturity Level | Keyword | Meaning |
|---------------|---------|---------|
| **Minimum** | SHALL | Required for all Solution Architecture Documents |
| **Recommended** | SHOULD | Required for production-grade designs |
| **Comprehensive** | MAY | For mature, enterprise-grade documentation |
