## Was ist ADS?

ADS ist ein **kostenloser, quelloffener, präskriptiver Standard**, der genau festlegt, welche Abschnitte, Inhalte und Qualitätskriterien ein Solution Architecture Document (SAD) enthalten MUSS. Er wird unter der [Creative Commons Attribution 4.0](https://creativecommons.org/licenses/by/4.0/) Lizenz veroffentlicht — jeder darf ihn nutzen, anpassen und weiterverbreiten, auch für kommerzielle Zwecke.

Er basiert auf — ist aber keine Kopie von — ISO/IEC/IEEE 42010, das ein abstraktes konzeptionelles Modell für Architekturbeschreibungen definiert. Wahrend ISO 42010 besagt *"eine Architekturbeschreibung MUSS Stakeholder und deren Anliegen identifizieren"*, legt ADS fest *"Abschnitt 2.1 MUSS eine Stakeholder-Register-Tabelle mit Spalten für Stakeholder, Rolle, zentrale Anliegen und relevante Architekturansichten enthalten."*

### Das Benennungsmodell

| Begriff | Bedeutung | In diesem Standard |
|---------|-----------|-------------------|
| **Architecture Description** | Der formale ISO 42010-Begriff für das Arbeitsergebnis | Der Standard selbst (ADS) orientiert sich an diesem Konzept |
| **Solution Architecture Document (SAD)** | Die in der Unternehmenspraxis gebrauchliche Bezeichnung für das Dokument. Das Akronym SAD stammt aus dem Rational Unified Process (RUP), wo es für "Software Architecture Document" stand. Dieser Standard verwendet "Solution", um den breiteren Umfang jenseits von Software widerzuspiegeln — einschliesslich Infrastruktur, Daten, Sicherheit und Betrieb. | Das Dokument, das Sie mit diesem Standard erstellen |
| **High Level Design (HLD)** | Der konzeptionelle Designinhalt | Abschnitte 3–4 (Architekturansichten + Qualitätsattribute) innerhalb des SAD |

## Geltungsbereich

Dieser Standard definiert die erforderliche und empfohlene Struktur für Solution Architecture Documents, die:

- **Organisationsunabhängig** sind — von jedem Unternehmen ohne Anderung übernehmbar
- **Plattformneutral** sind — anwendbar auf Cloud-, On-Premises-, Hybrid- und SaaS-Architekturen
- **Framework-konform** sind — rückverfolgbbar zu ISO/IEC 42010, dem 4+1 View Model und den AWS/Azure Well-Architected Frameworks
- **Schemagesteuert** sind — unterstutzt durch ein formales JSON Schema zur Validierung und Erzeugung in mehreren Formaten

## Über ISO/IEC 42010 hinaus

ADS *entspricht* dem konzeptionellen Modell von ISO 42010 und bietet gleichzeitig die konkrete, handlungsorientierte Struktur, die ISO 42010 bewusst den Implementierern überlasst. Es geht weit über ISO 42010 hinaus, indem es Folgendes einbezieht:

- **Qualitätsattribute aus den AWS-, Azure- und GCP Well-Architected Frameworks** — Betriebliche Exzellenz, Zuverlässigkeit, Leistungseffizienz, Kostenoptimierung und Nachhaltigkeit als querschnittliche Bewertungsebenen über alle Architekturansichten
- **Dokumentationstiefen mit RFC 2119-Anforderungsschlusselwortern** — jeder Abschnitt ist als SHALL (Minimum), SHOULD (Empfohlen) oder MAY (Umfassend) gekennzeichnet, was eine schrittweise Einführung von leichtgewichtigen Designs bis hin zu vollständiger Unternehmensdokumentation ermoglicht
- **Sicherheit als eigene Architekturansicht** — ISO 42010 schreibt keine spezifischen Blickwinkel vor; ADS erhebt Sicherheit zu einer erstklassigen Ansicht, die IAM, Verschlusselung, Bedrohungsmodellierung und Überwachung abdeckt
- **Lebenszyklusmanagement** — CI/CD-Pipelines, Dienstübergang, Release-Management, Betrieb, Wartbarkeit und Ausstiegsplanung
- **Entscheidungsfindung & Governance mit Compliance-Rückverfolgbarkeit** — Designrisiken, Leitplanken-Ausnahmen, Architecture Decision Records und eine Rückverfolgbarkeitsmatrix, die Designelemente den Standards zuordnet, die sie erfullen
- **JSON Schema für Multi-Format-Erzeugung** — ein formales Schema zur Validierung und Erzeugung von Solution Architecture Documents in JSON, YAML, Markdown oder Word
- **Herunterladbare Vorlagen** — sofort einsatzbereite leere Vorlagen in Markdown-, YAML- und JSON-Formaten

### Vergleich mit ISO/IEC 42010

| Aspekt | ISO/IEC/IEEE 42010:2022 | ADS (Dieser Standard) |
|--------|------------------------|------------------------|
| **Typ** | Meta-Standard (konzeptionelles Modell) | Präskriptiver Standard (Vorlagenebene) |
| **Schreibt Abschnitte vor?** | Nein — definiert nur Konzepte | Ja — definiert exakte Abschnitte, Tabellen und Inhalte |
| **Schreibt Qualitätskriterien vor?** | Nein | Ja — Qualitätsattribute aus AWS/Azure WAF |
| **Stellt Vorlagen bereit?** | Nein | Ja — Markdown-, YAML-, JSON-Vorlagen |
| **Definiert Dokumentationstiefen?** | Nein | Ja — Minimum, Empfohlen, Umfassend |
| **Beinhaltet Sicherheitsansicht?** | Nicht spezifisch | Ja — vollständige eigene Sicherheitsansicht |
| **Beinhaltet Lebenszyklus?** | Nein | Ja — CI/CD, Betrieb, Ausstiegsplanung |
| **JSON Schema?** | Nein | Ja — maschinenlesbare Validierung |

## Normative Referenzen

| Referenz | Titel | Verwendung in ADS |
|----------|-------|-------------------|
| ISO/IEC/IEEE 42010:2022 | Architecture Description | Konzeptionelles Modell: Stakeholder, Anliegen, Blickwinkel, Ansichten |
| Kruchten 4+1 | Architectural View Model | Sechs-Ansichten-Struktur (erweitert um Daten- und Sicherheitsansichten) |
| AWS Well-Architected | Well-Architected Framework | Definitionen und Bewertungskriterien für Qualitätsattribute |
| Azure Well-Architected | Well-Architected Framework | Definitionen von Qualitätsattributen mit Tradeoff-Dokumentation |
| GCP Architecture | Well-Architected Framework | Definitionen von Qualitätsattributen mit querschnittlichen Perspektiven |
| Oracle Cloud | Well-Architected Framework | Definitionen von Qualitätsattributen einschliesslich Distributed Cloud |
| IBM Cloud | Well-Architected Framework | Definitionen von Qualitätsattributen einschliesslich Hybrid und Portable |
| TOGAF 10th Edition | The Open Group Architecture Framework | Ausrichtung der Architekturdomanen |
| RFC 2119 / RFC 8174 | Requirement Level Key Words | SHALL / SHOULD / MAY Terminologie |

## Begriffe und Definitionen

Die Schlusselworter "SHALL", "SHOULD", "MAY" und "OPTIONAL" in diesem Standard sind wie in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) und [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174) beschrieben zu interpretieren:

| Dokumentationstiefe | Schlusselwort | Bedeutung |
|---------------------|---------------|-----------|
| **Minimum** | SHALL | Erforderlich für alle Solution Architecture Documents |
| **Empfohlen** | SHOULD | Erforderlich für produktionsreife Designs |
| **Umfassend** | MAY | Für ausgereiftes Unternehmensdokumentationsniveau |
