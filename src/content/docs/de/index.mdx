## Was ist ADS?

ADS ist ein **kostenloser, quelloffener, praskriptiver Standard**, der genau festlegt, welche Abschnitte, Inhalte und Qualitatskriterien ein Solution Architecture Document (SAD) enthalten MUSS. Er wird unter der [Creative Commons Attribution 4.0](https://creativecommons.org/licenses/by/4.0/) Lizenz veroffentlicht — jeder darf ihn nutzen, anpassen und weiterverbreiten, auch fur kommerzielle Zwecke.

Er basiert auf — ist aber keine Kopie von — ISO/IEC/IEEE 42010, das ein abstraktes konzeptionelles Modell fur Architekturbeschreibungen definiert. Wahrend ISO 42010 besagt *"eine Architekturbeschreibung MUSS Stakeholder und deren Anliegen identifizieren"*, legt ADS fest *"Abschnitt 2.1 MUSS eine Stakeholder-Register-Tabelle mit Spalten fur Stakeholder, Rolle, zentrale Anliegen und relevante Architekturansichten enthalten."*

### Das Benennungsmodell

| Begriff | Bedeutung | In diesem Standard |
|---------|-----------|-------------------|
| **Architecture Description** | Der formale ISO 42010-Begriff fur das Arbeitsergebnis | Der Standard selbst (ADS) orientiert sich an diesem Konzept |
| **Solution Architecture Document (SAD)** | Die in der Unternehmenspraxis gebrauchliche Bezeichnung fur das Dokument. Das Akronym SAD stammt aus dem Rational Unified Process (RUP), wo es fur "Software Architecture Document" stand. Dieser Standard verwendet "Solution", um den breiteren Umfang jenseits von Software widerzuspiegeln — einschliesslich Infrastruktur, Daten, Sicherheit und Betrieb. | Das Dokument, das Sie mit diesem Standard erstellen |
| **High Level Design (HLD)** | Der konzeptionelle Designinhalt | Abschnitte 3–4 (Architekturansichten + Qualitatsattribute) innerhalb des SAD |

## Geltungsbereich

Dieser Standard definiert die erforderliche und empfohlene Struktur fur Solution Architecture Documents, die:

- **Organisationsunabhangig** sind — von jedem Unternehmen ohne Anderung ubernehmbar
- **Plattformneutral** sind — anwendbar auf Cloud-, On-Premises-, Hybrid- und SaaS-Architekturen
- **Framework-konform** sind — ruckverfolgbar zu ISO/IEC 42010, dem 4+1 View Model und den AWS/Azure Well-Architected Frameworks
- **Schemagesteuert** sind — unterstutzt durch ein formales JSON Schema zur Validierung und Erzeugung in mehreren Formaten

## Uber ISO/IEC 42010 hinaus

ADS *entspricht* dem konzeptionellen Modell von ISO 42010 und bietet gleichzeitig die konkrete, handlungsorientierte Struktur, die ISO 42010 bewusst den Implementierern uberlasst. Es geht weit uber ISO 42010 hinaus, indem es Folgendes einbezieht:

- **Qualitatsattribute aus den AWS-, Azure- und GCP Well-Architected Frameworks** — Betriebliche Exzellenz, Zuverlassigkeit, Leistungseffizienz, Kostenoptimierung und Nachhaltigkeit als querschnittliche Bewertungsebenen uber alle Architekturansichten
- **Dokumentationstiefen mit RFC 2119-Anforderungsschlusselwortern** — jeder Abschnitt ist als SHALL (Minimum), SHOULD (Empfohlen) oder MAY (Umfassend) gekennzeichnet, was eine schrittweise Einfuhrung von leichtgewichtigen Designs bis hin zu vollstandiger Unternehmensdokumentation ermoglicht
- **Sicherheit als eigene Architekturansicht** — ISO 42010 schreibt keine spezifischen Blickwinkel vor; ADS erhebt Sicherheit zu einer erstklassigen Ansicht, die IAM, Verschlusselung, Bedrohungsmodellierung und Uberwachung abdeckt
- **Lebenszyklusmanagement** — CI/CD-Pipelines, Dienstubergang, Release-Management, Betrieb, Wartbarkeit und Ausstiegsplanung
- **Entscheidungsfindung & Governance mit Compliance-Ruckverfolgbarkeit** — Designrisiken, Leitplanken-Ausnahmen, Architecture Decision Records und eine Ruckverfolgbarkeitsmatrix, die Designelemente den Standards zuordnet, die sie erfullen
- **JSON Schema fur Multi-Format-Erzeugung** — ein formales Schema zur Validierung und Erzeugung von Solution Architecture Documents in JSON, YAML, Markdown oder Word
- **Herunterladbare Vorlagen** — sofort einsatzbereite leere Vorlagen in Markdown-, YAML- und JSON-Formaten

### Vergleich mit ISO/IEC 42010

| Aspekt | ISO/IEC/IEEE 42010:2022 | ADS (Dieser Standard) |
|--------|------------------------|------------------------|
| **Typ** | Meta-Standard (konzeptionelles Modell) | Praskriptiver Standard (Vorlagenebene) |
| **Schreibt Abschnitte vor?** | Nein — definiert nur Konzepte | Ja — definiert exakte Abschnitte, Tabellen und Inhalte |
| **Schreibt Qualitatskriterien vor?** | Nein | Ja — Qualitatsattribute aus AWS/Azure WAF |
| **Stellt Vorlagen bereit?** | Nein | Ja — Markdown-, YAML-, JSON-Vorlagen |
| **Definiert Dokumentationstiefen?** | Nein | Ja — Minimum, Empfohlen, Umfassend |
| **Beinhaltet Sicherheitsansicht?** | Nicht spezifisch | Ja — vollstandige eigene Sicherheitsansicht |
| **Beinhaltet Lebenszyklus?** | Nein | Ja — CI/CD, Betrieb, Ausstiegsplanung |
| **JSON Schema?** | Nein | Ja — maschinenlesbare Validierung |

## Normative Referenzen

| Referenz | Titel | Verwendung in ADS |
|----------|-------|-------------------|
| ISO/IEC/IEEE 42010:2022 | Architecture Description | Konzeptionelles Modell: Stakeholder, Anliegen, Blickwinkel, Ansichten |
| Kruchten 4+1 | Architectural View Model | Sechs-Ansichten-Struktur (erweitert um Daten- und Sicherheitsansichten) |
| AWS Well-Architected | Well-Architected Framework | Definitionen und Bewertungskriterien fur Qualitatsattribute |
| Azure Well-Architected | Well-Architected Framework | Definitionen von Qualitatsattributen mit Tradeoff-Dokumentation |
| GCP Architecture | Well-Architected Framework | Definitionen von Qualitatsattributen mit querschnittlichen Perspektiven |
| Oracle Cloud | Well-Architected Framework | Definitionen von Qualitatsattributen einschliesslich Distributed Cloud |
| IBM Cloud | Well-Architected Framework | Definitionen von Qualitatsattributen einschliesslich Hybrid und Portable |
| TOGAF 10th Edition | The Open Group Architecture Framework | Ausrichtung der Architekturdomanen |
| RFC 2119 / RFC 8174 | Requirement Level Key Words | SHALL / SHOULD / MAY Terminologie |

## Begriffe und Definitionen

Die Schlusselworter "SHALL", "SHOULD", "MAY" und "OPTIONAL" in diesem Standard sind wie in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) und [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174) beschrieben zu interpretieren:

| Dokumentationstiefe | Schlusselwort | Bedeutung |
|---------------------|---------------|-----------|
| **Minimum** | SHALL | Erforderlich fur alle Solution Architecture Documents |
| **Empfohlen** | SHOULD | Erforderlich fur produktionsreife Designs |
| **Umfassend** | MAY | Fur ausgereiftes Unternehmensdokumentationsniveau |
