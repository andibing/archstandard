## Zweck

Dieser Standard definiert die erforderliche Struktur und den Inhalt für **Solution Architecture Documents (SADs)**. Die Architekturansichten und Qualitätsattribute innerhalb eines SAD bilden das **High Level Design (HLD)** der Lösung.

Ein Solution Architecture Document, das diesem Standard entspricht, erfüllt die folgenden Funktionen:

1. **Kommunikation** — Bietet ein gemeinsames Verständnis der Lösungsarchitektur für alle Stakeholder
2. **Governance** — Ermöglicht dem Architektur-Governance-Prozess, das Design anhand von Qualitätskriterien zu bewerten
3. **Rückverfolgbarkeit** — Stellt Verbindungen zwischen Designentscheidungen, Geschäftsanforderungen, Qualitätsattributen und geltenden Standards her
4. **Referenz** — Dient als lebendes Dokument, das die aktuelle Lösungsarchitektur akkurat beschreibt
5. **Nachweis** — Belegt, dass die Lösung gegenüber branchenweit anerkannten Frameworks gut architekturiert ist

## Geltungsbereich

Dieser Standard gilt für die Dokumentation jeder technologischen Lösungsarchitektur, unabhängig von:

- **Bereitstellungsmodell** — Cloud-nativ, On-Premises, Hybrid, Multi-Cloud oder SaaS
- **Skalierung** — von Einzelanwendungen bis zu Multi-Anwendungs-Okosystemen
- **Branche** — der Standard ist organisationsunabhängig und branchenneutral
- **Lebenszyklusphase** — Neubauten, Migrationen, Modernisierungen oder inkrementelle Änderungen

## Zielgruppe

Dieser Standard ist für die Nutzung durch folgende Rollen vorgesehen:

| Rolle | Primare Abschnitte |
|-------|-------------------|
| **Solution Architects** | Alle Abschnitte — primare Autoren von SADs |
| **Enterprise Architects** | Framework-Ausrichtung, Qualitätsattribute, Governance |
| **Security Architects** | Abschnitt 3.5 (Sicherheitsansicht), Qualitätsattribute |
| **Infrastruktur-Ingenieure** | Abschnitt 3.3 (Physische Ansicht), Qualitätsattribute |
| **Entwicklungsleiter** | Abschnitt 3.1 (Logische Ansicht), Abschnitt 5 (Lebenszyklus) |
| **Architektur-Governance** | Alle Abschnitte — als Prüfungs- und Governance-Rahmenwerk (z.B. ARB, Design Authority) |

## Dokumentstruktur

Ein Solution Architecture Document, das ADS entspricht, ist in drei Ebenen gegliedert:

### Ebene 1 — Architekturansichten (Abschnitte 3.1 – 3.6)

Sechs komplementare Ansichten beschreiben die Architektur aus unterschiedlichen Stakeholder-Perspektiven, ausgerichtet am 4+1 Architectural View Model von Kruchten (erweitert um eigene Daten- und Sicherheitsansichten) und dem ISO/IEC 42010 Viewpoint-Konzept.

Gemeinsam bilden diese Ansichten das **High Level Design (HLD)** der Lösung:

| Ansicht | Blickwinkel | Primare Stakeholder |
|---------|-------------|---------------------|
| **3.1 Logische Ansicht** | Anwendungsarchitektur, Komponenten, Muster | Architekten, Entwickler |
| **3.2 Integrations- & Datenflussansicht** | Datenflusse, Integrationen, Schnittstellen | Integratoren, Architekten |
| **3.3 Physische Ansicht** | Bereitstellung, Hosting, Netzwerk, Umgebungen | Infrastruktur, DevOps |
| **3.4 Datenansicht** | Datenspeicher, Klassifizierung, Datenschutz, Lebenszyklus | Datenarchitekten, Compliance |
| **3.5 Sicherheitsansicht** | IAM, Verschlüsselung, Überwachung, Bedrohungsmodell | Sicherheit, CISO, Compliance |
| **3.6 Szenarien** | Wichtige Anwendungsfalle, Architecture Decision Records | Alle Stakeholder |

Keine einzelne Ansicht liefert eine vollständige Beschreibung der Architektur. Zusammen bieten die sechs Ansichten ein ganzheitliches Bild, das alle Stakeholder-Anliegen adressiert.

### Ebene 2 — Qualitätsattribute (Abschnitte 4.1 – 4.5)

Funf querschnittliche Qualitätsattribute werden **über** alle Architekturansichten hinweg bewertet. Diese stammen aus den AWS und Azure Well-Architected Frameworks:

| Qualitätsattribut | Schwerpunkt |
|-------------------|-------------|
| **4.1 Betriebliche Exzellenz** | Beobachtbarkeit, Überwachung, Betriebsverfahren |
| **4.2 Zuverlässigkeit & Resilienz** | DR, Skalierbarkeit, Fehlertoleranz, Backup und Wiederherstellung |
| **4.3 Leistungseffizienz** | Leistungsanforderungen, Ressourcenoptimierung |
| **4.4 Kostenoptimierung** | Kostenanalyse, FinOps-Praktiken |
| **4.5 Nachhaltigkeit** | Energieeffizienz, CO2-Auswirkungen, Ressourceneffizienz |

### Ebene 3 — Lebenszyklus & Governance (Abschnitte 5 – 7)

Unterstutzende Abschnitte behandeln, wie die Lösung entwickelt, betrieben, gesteuert und ausser Betrieb genommen wird.

## Dokumentationstiefen

Jedem Abschnitt in diesem Standard ist eine Dokumentationstiefe zugewiesen — <span class="maturity-indicator maturity-minimum">Minimum</span> (SHALL), <span class="maturity-indicator maturity-recommended">Empfohlen</span> (SHOULD) oder <span class="maturity-indicator maturity-comprehensive">Umfassend</span> (MAY) — die festlegt, wann er ausgefüllt werden muss. Dies ermöglicht eine schrittweise Einführung: Beginnen Sie mit Minimum für frühphasige Designs und erweitern Sie auf Umfassend für kritische und regulierte Systeme.

Siehe [Konformität und Anwendung](/v1/de/standard/how-to-use/) für die vollständige Tiefentabelle, Terminologiedefinitionen und die Zuordnung zu Governance-Gates.

## Konformität

Ein Solution Architecture Document **entspricht** diesem Standard, wenn:

1. Alle als <span class="maturity-indicator maturity-minimum">Minimum</span> gekennzeichneten Abschnitte ausgefüllt sind
2. Die Dokumentstruktur der hierin definierten Abschnittsnummerierung und -hierarchie folgt
3. Architekturansichten die in Abschnitt 2 identifizierten Stakeholder-Anliegen adressieren
4. Qualitätsattribute als querschnittliche Bewertungsebenen über die Architekturansichten bewertet werden
5. Das Dokument gegen das JSON Schema (ADS Schema v1.0.0) validiert wird, falls maschinenlesbare Ausgabe erforderlich ist

Ein Dokument, das zusätzlich alle als <span class="maturity-indicator maturity-recommended">Empfohlen</span> gekennzeichneten Abschnitte vervollständigt, erreicht **vollständige Konformität**.
