## Zweck

Dieser Standard definiert die erforderliche Struktur und den Inhalt fur **Solution Architecture Documents (SADs)**. Die Architekturansichten und Qualitatsattribute innerhalb eines SAD bilden das **High Level Design (HLD)** der Losung.

Ein Solution Architecture Document, das diesem Standard entspricht, erfullt die folgenden Funktionen:

1. **Kommunikation** — Bietet ein gemeinsames Verstandnis der Losungsarchitektur fur alle Stakeholder
2. **Governance** — Ermoglicht dem Architektur-Governance-Prozess, das Design anhand von Qualitatskriterien zu bewerten
3. **Ruckverfolgbarkeit** — Stellt Verbindungen zwischen Designentscheidungen, Geschaftsanforderungen, Qualitatsattributen und geltenden Standards her
4. **Referenz** — Dient als lebendes Dokument, das die aktuelle Losungsarchitektur akkurat beschreibt
5. **Nachweis** — Belegt, dass die Losung gegenuber branchenweit anerkannten Frameworks gut architekturiert ist

## Geltungsbereich

Dieser Standard gilt fur die Dokumentation jeder technologischen Losungsarchitektur, unabhangig von:

- **Bereitstellungsmodell** — Cloud-nativ, On-Premises, Hybrid, Multi-Cloud oder SaaS
- **Skalierung** — von Einzelanwendungen bis zu Multi-Anwendungs-Okosystemen
- **Branche** — der Standard ist organisationsunabhangig und branchenneutral
- **Lebenszyklusphase** — Neubauten, Migrationen, Modernisierungen oder inkrementelle Anderungen

## Zielgruppe

Dieser Standard ist fur die Nutzung durch folgende Rollen vorgesehen:

| Rolle | Primare Abschnitte |
|-------|-------------------|
| **Solution Architects** | Alle Abschnitte — primare Autoren von SADs |
| **Enterprise Architects** | Framework-Ausrichtung, Qualitatsattribute, Governance |
| **Security Architects** | Abschnitt 3.5 (Sicherheitsansicht), Qualitatsattribute |
| **Infrastruktur-Ingenieure** | Abschnitt 3.3 (Physische Ansicht), Qualitatsattribute |
| **Entwicklungsleiter** | Abschnitt 3.1 (Logische Ansicht), Abschnitt 5 (Lebenszyklus) |
| **Architektur-Governance** | Alle Abschnitte — als Prufungs- und Governance-Rahmenwerk (z.B. ARB, Design Authority) |

## Dokumentstruktur

Ein Solution Architecture Document, das ADS entspricht, ist in drei Ebenen gegliedert:

### Ebene 1 — Architekturansichten (Abschnitte 3.1 – 3.6)

Sechs komplementare Ansichten beschreiben die Architektur aus unterschiedlichen Stakeholder-Perspektiven, ausgerichtet am 4+1 Architectural View Model von Kruchten (erweitert um eigene Daten- und Sicherheitsansichten) und dem ISO/IEC 42010 Viewpoint-Konzept.

Gemeinsam bilden diese Ansichten das **High Level Design (HLD)** der Losung:

| Ansicht | Blickwinkel | Primare Stakeholder |
|---------|-------------|---------------------|
| **3.1 Logische Ansicht** | Anwendungsarchitektur, Komponenten, Muster | Architekten, Entwickler |
| **3.2 Integrations- & Datenflussansicht** | Datenflusse, Integrationen, Schnittstellen | Integratoren, Architekten |
| **3.3 Physische Ansicht** | Bereitstellung, Hosting, Netzwerk, Umgebungen | Infrastruktur, DevOps |
| **3.4 Datenansicht** | Datenspeicher, Klassifizierung, Datenschutz, Lebenszyklus | Datenarchitekten, Compliance |
| **3.5 Sicherheitsansicht** | IAM, Verschlusselung, Uberwachung, Bedrohungsmodell | Sicherheit, CISO, Compliance |
| **3.6 Szenarien** | Wichtige Anwendungsfalle, Architecture Decision Records | Alle Stakeholder |

Keine einzelne Ansicht liefert eine vollstandige Beschreibung der Architektur. Zusammen bieten die sechs Ansichten ein ganzheitliches Bild, das alle Stakeholder-Anliegen adressiert.

### Ebene 2 — Qualitatsattribute (Abschnitte 4.1 – 4.5)

Funf querschnittliche Qualitatsattribute werden **uber** alle Architekturansichten hinweg bewertet. Diese stammen aus den AWS und Azure Well-Architected Frameworks:

| Qualitatsattribut | Schwerpunkt |
|-------------------|-------------|
| **4.1 Betriebliche Exzellenz** | Beobachtbarkeit, Uberwachung, Betriebsverfahren |
| **4.2 Zuverlassigkeit & Resilienz** | DR, Skalierbarkeit, Fehlertoleranz, Backup und Wiederherstellung |
| **4.3 Leistungseffizienz** | Leistungsanforderungen, Ressourcenoptimierung |
| **4.4 Kostenoptimierung** | Kostenanalyse, FinOps-Praktiken |
| **4.5 Nachhaltigkeit** | Energieeffizienz, CO2-Auswirkungen, Ressourceneffizienz |

### Ebene 3 — Lebenszyklus & Governance (Abschnitte 5 – 7)

Unterstutzende Abschnitte behandeln, wie die Losung entwickelt, betrieben, gesteuert und ausser Betrieb genommen wird.

## Dokumentationstiefen

Jedem Abschnitt in diesem Standard ist eine Dokumentationstiefe zugewiesen — <span class="maturity-indicator maturity-minimum">Minimum</span> (SHALL), <span class="maturity-indicator maturity-recommended">Empfohlen</span> (SHOULD) oder <span class="maturity-indicator maturity-comprehensive">Umfassend</span> (MAY) — die festlegt, wann er ausgefullt werden muss. Dies ermoglicht eine schrittweise Einfuhrung: Beginnen Sie mit Minimum fur fruhphasige Designs und erweitern Sie auf Umfassend fur kritische und regulierte Systeme.

Siehe [Konformitat und Anwendung](/de/standard/how-to-use/) fur die vollstandige Tiefentabelle, Terminologiedefinitionen und die Zuordnung zu Governance-Gates.

## Konformitat

Ein Solution Architecture Document **entspricht** diesem Standard, wenn:

1. Alle als <span class="maturity-indicator maturity-minimum">Minimum</span> gekennzeichneten Abschnitte ausgefullt sind
2. Die Dokumentstruktur der hierin definierten Abschnittsnummerierung und -hierarchie folgt
3. Architekturansichten die in Abschnitt 2 identifizierten Stakeholder-Anliegen adressieren
4. Qualitatsattribute als querschnittliche Bewertungsebenen uber die Architekturansichten bewertet werden
5. Das Dokument gegen das JSON Schema (ADS Schema v1.0.0) validiert wird, falls maschinenlesbare Ausgabe erforderlich ist

Ein Dokument, das zusatzlich alle als <span class="maturity-indicator maturity-recommended">Empfohlen</span> gekennzeichneten Abschnitte vervollstandigt, erreicht **vollstandige Konformitat**.
