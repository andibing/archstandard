## Ansichtenmodell

Der ADS organisiert die Lösungsarchitektur in **sechs komplementare Ansichten**, die jeweils eine bestimmte Menge von Stakeholder-Anliegen adressieren. Dieser Ansatz ist ausgerichtet am 4+1 Architectural View Model von Kruchten, erweitert um eigene Daten- und Sicherheitsansichten.

Keine einzelne Ansicht liefert ein vollständiges Bild der Architektur. Zusammen bieten die Ansichten eine ganzheitliche Beschreibung, die von allen Stakeholdern verstanden werden kann.

## Die sechs Ansichten

| # | Ansicht | Basiert auf | Primare Stakeholder | Beschreibt |
|---|---------|-------------|---------------------|-----------|
| 3.1 | **Logische Ansicht** | 4+1 Logical | Architekten, Entwickler | Anwendungskomponenten, Dienste, Muster |
| 3.2 | **Integrations- & Datenflussansicht** | 4+1 Process (angepasst) | Integratoren, Architekten | Datenflusse, Integrationen, Schnittstellen |
| 3.3 | **Physische Ansicht** | 4+1 Physical | Infrastruktur, DevOps, Cloud | Gesamte Bereitstellungsinfrastruktur — physisch, virtuell, Cloud und Serverless |
| 3.4 | **Datenansicht** | RM-ODP Information | Datenarchitekten, DBA, Compliance | Datenspeicher, Klassifizierung, Datenschutz, Lebenszyklus |
| 3.5 | **Sicherheitsansicht** | Erweitert | Sicherheit, CISO, Compliance | IAM, Verschlüsselung, Überwachung, Bedrohungsmodell |
| 3.6 | **Szenarien** | 4+1 +1 | Alle Stakeholder | Anwendungsfalle, Architecture Decision Records |

## Korrespondenzen zwischen Ansichten

Ansichten sind nicht unabhängig — sie teilen Korrespondenzen (Beziehungen zwischen Elementen über Ansichten hinweg). Wesentliche Korrespondenzen umfassen:

- **Logische Komponenten** (3.1) werden **bereitgestellt auf** physischer Infrastruktur (3.3)
- **Datenspeicher** (3.4) werden **gehostet innerhalb** der physischen Umgebung (3.3)
- **Integrationsflusse** (3.2) durchlaufen **Netzwerkpfade**, die in der Physischen Ansicht (3.3) definiert sind
- **Sicherheitskontrollen** (3.5) werden **angewandt auf** Komponenten, Daten und Infrastruktur über alle Ansichten
- **Szenarien** (3.6) **validieren** das Design über alle Ansichten

Wenn Sie eine Ansicht dokumentieren, verweisen Sie auf verwandte Elemente in anderen Ansichten, um die Rückverfolgbarkeit zu gewährleisten.

## Diagrammkonventionen

Jede Ansicht sollte mindestens ein Diagramm enthalten. Befolgen Sie diese Konventionen:

- Verwenden Sie eine konsistente Notation (z.B. UML, ArchiMate, C4 Model oder einfache Blockdiagramme)
- Beschriften Sie alle Komponenten, Verbindungen und Grenzen deutlich
- Zeigen Sie Vertrauensgrenzen in sicherheitsrelevanten Diagrammen
- Fugen Sie eine Legende hinzu, wenn die Notation nicht selbsterklärend ist
- Stellen Sie sicher, dass Diagramme in einer Auflösung vorliegen, die das Lesen von Text ermöglicht
