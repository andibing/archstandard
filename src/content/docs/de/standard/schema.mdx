## Überblick

Der ADS definiert ein formales **JSON Schema** (v1.0.0), das Folgendes ermöglicht:

- **Validierung** - Solution Architecture Documents gegen die Standardstruktur validieren
- **Generierung** - Leere Vorlagen in mehreren Formaten generieren (JSON, YAML, Markdown, Word)
- **Tooling** - Werkzeuge erstellen, die Solution Architecture Documents programmatisch lesen, schreiben und verarbeiten
- **Erweiterbarkeit** - Organisationsspezifische Abschnitte über `customSections` und `organisationProfile` hinzufügen

## Schema-Speicherort

Die Schema-Datei befindet sich unter [`/schema/ads.schema.json`](https://github.com/andibing/archstandard/blob/main/schema/ads.schema.json) im Repository.

**Schema-ID:** `https://archstandard.org/schema/v1.0.0/ads.schema.json`

## Schema-Struktur

Das Schema bildet direkt die Abschnitte des Standards ab:

```
ads.schema.json
├── schemaVersion          "1.0.0"
├── documentControl        Abschnitt 0: Metadaten, Historie, Mitwirkende
├── executiveSummary       Abschnitt 1: Überblick, Kontext, Scope, Entscheidungen
├── stakeholders           Abschnitt 2: Register, Concerns, Compliance
├── architecturalViews     Abschnitt 3
│   ├── logicalView        3.1: Komponenten, Services, Muster
│   ├── integrationView     3.2: Konnektivität, Integrationen, APIs
│   ├── physicalView       3.3: Hosting, Compute, Netzwerk, Umgebungen
│   ├── dataView           3.4: Datenspeicher, Klassifizierung, Datenschutz
│   ├── securityView       3.5: IAM, Verschlüsselung, Überwachung
│   └── scenarios          3.6: Anwendungsfälle, ADRs
├── qualityAttributes      Abschnitt 4: Qualitätsattribute + Kompromisse
├── lifecycleManagement    Abschnitt 5: SDLC, Betrieb, Exit-Planung
├── riskGovernance         Abschnitt 6: Einschränkungen, Risiken, Entscheidungen, Compliance
├── appendices             Abschnitt 7: Glossar, Referenzen, Genehmigungen
├── organisationProfile    Organisationsspezifische Konfiguration
└── customSections[]       Erweiterungspunkt für zusätzliche Abschnitte
```

## Wichtige Schema-Funktionen

### Documentation Depths

Abschnitte verwenden ein `maturityLevel`-Enum mit drei Werten. Hinweis: Das Schema verwendet den Eigenschaftsnamen `maturityLevel`, während die Prosa des Standards dieses Konzept als „Documentation Depth" bezeichnet — sie bedeuten dasselbe:

```json
{ "type": "string", "enum": ["minimum", "recommended", "comprehensive"] }
```

### Quality Attribute References

Jeder Abschnitt kann auf zutreffende Qualitätsattribute verweisen:

```json
{
  "qualityAttributeRefs": ["reliability", "performance", "cost-optimisation"]
}
```

### Organisation Profile

Die `organisationProfile`-Erweiterung ermöglicht die Zuordnung des generischen Standards zu organisationsspezifischen Tools und Governance:

```json
{
  "organisationProfile": {
    "organisationName": "Acme Corp",
    "tooling": {
      "cmdb": "ServiceNow",
      "secretStore": "HashiCorp Vault",
      "monitoring": "Datadog"
    },
    "governanceGates": [
      {
        "gateName": "Dev/Test Review",
        "requiredMaturityLevel": "minimum",
        "requiredSections": ["documentControl", "executiveSummary", "architecturalViews.logicalView"]
      },
      {
        "gateName": "Production Approval",
        "requiredMaturityLevel": "recommended"
      }
    ]
  }
}
```

### Custom Sections

Organisationsspezifische Abschnitte hinzufügen, ohne das Kernschema zu ändern:

```json
{
  "customSections": [
    {
      "id": "cloud-governance",
      "title": "Cloud Governance",
      "description": "Organisation-specific cloud governance requirements",
      "content": "...",
      "maturityLevel": "recommended",
      "parentSection": "physicalView"
    }
  ]
}
```

## Pflichtfelder vs. optionale Felder

Das Schema erzwingt nur die **Minimum**-Documentation-Depth als Pflichtfelder:

| Pflichtfeld | Abschnitt |
|------------|-----------|
| `schemaVersion` | Root |
| `documentControl.metadata` | 0. Dokumentenlenkung |
| `executiveSummary.solutionOverview` | 1. Zusammenfassung |
| `architecturalViews` | 3. Architekturansichten |
| `riskGovernance.risks` | 6. Entscheidungsfindung & Governance |

Alle anderen Felder sind optional, was eine schrittweise Einführung von Minimum bis Comprehensive Documentation Depths ermöglicht.

## Validierungsbeispiel

Validieren Sie ein SAD gegen das Schema mit lokalen oder entfernten Referenzen:

```bash
# Using ajv-cli (local schema)
npx ajv validate -s schema/ads.schema.json -d my-sad.json

# Using ajv-cli (remote schema)
npx ajv validate -s https://archstandard.org/schema/v1.0.0/ads.schema.json -d my-sad.json

# Using Python jsonschema
pip install jsonschema
python -m jsonschema -i my-sad.json schema/ads.schema.json
```

Das Schema wird gehostet unter: [`https://archstandard.org/schema/v1.0.0/ads.schema.json`](https://archstandard.org/schema/v1.0.0/ads.schema.json)

## Tools mit dem Schema erstellen

Das JSON Schema ist für die Nutzung durch Tooling konzipiert — Formular-Builder, Validatoren, Konverter und SAD-Verwaltungsplattformen. Benutzerdefinierte Erweiterungseigenschaften erleichtern dies:

| Erweiterung | Zweck | Beispiel |
|------------|-------|---------|
| `x-ads-section` | Abschnittsnummer | `"3.1"`, `"6.3"` |
| `x-ads-title` | Anzeigename | `"Logical View"`, `"Risks"` |
| `x-ads-depth` | Documentation Depth | `"minimum"`, `"recommended"`, `"comprehensive"` |

Diese Erweiterungen sind auf jedem wichtigen Abschnitt und Unterabschnitt im Schema vorhanden. Ein Tool kann den Schema-Baum durchlaufen und diese Eigenschaften verwenden, um:

- **Formulare zu generieren** mit korrekter Abschnittsgruppierung, Labels und Depth-Indikatoren
- **Abschnitte ein-/auszublenden** basierend auf der vom Benutzer gewählten Documentation Depth
- **Vollständigkeit zu validieren** gegen ein bestimmtes Depth-Level
- **Zu exportieren** nach JSON, YAML, Markdown oder DOCX

```javascript
// Example: fetch the schema and list all sections
const schema = await fetch('https://archstandard.org/schema/v1.0.0/ads.schema.json').then(r => r.json());

for (const [key, def] of Object.entries(schema.$defs)) {
  if (def['x-ads-section']) {
    console.log(`${def['x-ads-section']} ${def['x-ads-title']} [${def['x-ads-depth']}]`);
  }
}
```
